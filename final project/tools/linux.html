Linux

The Security Analyst's Operating System

Linux is the backbone of cybersecurity. Most servers, firewalls, SIEMs, and security tools run on Linux. Kali, Parrot, and Ubuntu are standard in SOCs and penetration testing. Understanding the Linux command line, file system, permissions, and networking stack is non-negotiable for blue team and red team roles.

Why Security Runs on Linux

- Transparency: Open source means you can audit every line of code. No hidden backdoors
- Tool Availability: Nmap, Wireshark, Metasploit, Suricata all developed for Linux first
- Command Line Power: Pipe, grep, awk, sed let you process gigabytes of logs in seconds
- Stability: Servers run for years without reboot. Critical for 24/7 SOC monitoring
- Permissions Model: Granular user/group controls prevent privilege escalation
- Containerization: Docker/K8s security tooling is Linux-native

Essential Security Commands

1. Network investigation:

# Show all listening ports and processes
sudo netstat -tulpn

# Real-time traffic capture
sudo tcpdump -i eth0 -w capture.pcap

# Check for suspicious connections
ss -tunap | grep ESTAB

First commands to run during incident response to find C2 beacons and open backdoors.

2. Log analysis:

# Find failed SSH logins
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -nr

# Monitor auth logs in real-time
tail -f /var/log/auth.log

# Search for IOC in all logs
grep -r "192.168.1.100" /var/log/

Parse millions of log lines instantly. This is why SOC analysts live in the terminal.

3. File integrity & forensics:

# Hash a file to check for tampering
sha256sum /etc/passwd

# Find SUID binaries - common privesc vector
find / -perm -4000 -type f 2>/dev/null

# Check for recently modified files
find /var/www -type f -mtime -1

Critical for detecting web shells and backdoored binaries after a breach.

Defensive Applications

Hardening Linux is day-one work for blue team. Disable root SSH, configure iptables/ufw, set up fail2ban, audit SUID files, and ship logs to a SIEM. Tools like Lynis and OpenSCAP automate compliance checks. In incident response, you’ll live in `/proc`, `/var/log`, and use `lsof` and `ps` to hunt malicious processes that don’t show up in `top`.

My Experience

I use bash scripting to automate log parsing from Security Onion, and write Python scripts that run as systemd services for continuous monitoring.
